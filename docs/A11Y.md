# Rally Accessibility Audit (WCAG 2.1 AA)

Scope: the product app shell (`src/App.jsx`), shared primitives
(`src/components/UI.jsx`), the design tokens (`src/index.css`), the command
palette (`src/components/CommandK.jsx`), and the coming-soon gate
(`src/gate/ComingSoon.jsx`). Enterprise buyers run automated (axe / Lighthouse)
plus manual keyboard + screen-reader passes; the findings below are ordered by
how likely they are to block a procurement checklist.

Two additive helpers ship alongside this audit and are referenced throughout:

- `src/components/SkipLink.jsx` - skip-to-content link.
- `src/lib/a11y.js` - `useFocusTrap`, `useEscapeKey`, `useAnnouncer` / `announce`,
  `usePrefersReducedMotion` / `prefersReducedMotion`, `getFocusable`.

No existing file was edited. Every fix below is a small, surgical change the main
thread applies.

---

## Priority 1 - blockers (fail an automated or basic keyboard audit)

### P1.1 No visible focus indicator anywhere
- Where: `src/index.css` (whole file). Only `.input/.select/.textarea:focus`
  (lines 176-179) draw a ring. `.btn` (line 111), `NavLink` rows, `.link`,
  clickable rows, tabs, and the switch have no `:focus-visible` style, and many
  set `outline: none` implicitly via the reset.
- Impact: keyboard-only users cannot tell where focus is. This is the single
  most common axe/Lighthouse failure and a guaranteed manual-audit reject.
- Fix (main thread, `src/index.css`, additive rule):
  ```css
  :where(a, button, [role="button"], input, select, textarea, [tabindex]):focus-visible {
    outline: 2px solid var(--accent);
    outline-offset: 2px;
    border-radius: var(--r-sm);
  }
  .btn:focus-visible { box-shadow: 0 0 0 3px var(--accent-50), 0 0 0 5px var(--accent); }
  ```
- WCAG: 2.4.7 Focus Visible.

### P1.2 Modal has no focus trap, no dialog semantics, no focus restore
- Where: `src/components/UI.jsx` `Modal`, lines 123-146. It handles Escape and
  locks body scroll (lines 124-130) but: focus stays on the trigger behind the
  overlay, Tab escapes to the page underneath, there is no
  `role="dialog"`/`aria-modal`, no label association, and focus is not returned
  to the trigger on close.
- Fix: wrap with `useFocusTrap` + `useEscapeKey` from `src/lib/a11y.js`.
  ```jsx
  import { useFocusTrap, useEscapeKey } from '../lib/a11y';
  const panelRef = useRef(null);
  useFocusTrap(panelRef, open);
  useEscapeKey(onClose, open);
  // backdrop:
  <div role="presentation" onClick={onClose} ...>
    <div ref={panelRef} role="dialog" aria-modal="true"
         aria-labelledby={title ? 'modal-title' : undefined} ...>
      <h4 id="modal-title" ...>{title}</h4>
  ```
  `useFocusTrap` already restores focus to the trigger on unmount.
- WCAG: 2.1.2 No Keyboard Trap (inverse), 2.4.3 Focus Order, 4.1.2 Name/Role/Value.

### P1.3 Command palette is mouse-only for its results and has no combobox semantics
- Where: `src/components/CommandK.jsx`. Result rows are `<div onClick>` with
  `onMouseEnter` (lines 83-96) - not reachable or activatable by keyboard other
  than the Arrow/Enter handler on the input (lines 60-65), and the visual
  "active" row is not exposed to screen readers. The overlay `<div>` (line 72)
  has no dialog role, no focus trap.
- Impact: a screen-reader user hears an unlabeled text field and no results.
- Fix: add combobox/listbox semantics and wire `aria-activedescendant`.
  ```jsx
  // input:
  role="combobox" aria-expanded="true" aria-controls="cmdk-list"
  aria-activedescendant={`cmdk-opt-${active}`} aria-autocomplete="list"
  aria-label="Search contacts, companies, deals, or jump to a page"
  // list container: role="listbox" id="cmdk-list"
  // each row: role="option" id={`cmdk-opt-${i}`} aria-selected={i === active}
  ```
  Wrap the panel in `useFocusTrap(panelRef, open)` and keep Escape (already
  present). No need to make each row tabbable - the combobox pattern keeps focus
  on the input and moves `aria-activedescendant`.
- WCAG: 4.1.2, 2.1.1 Keyboard.

### P1.4 No skip link / no main landmark target
- Where: `src/App.jsx`. `<main className="rl-content">` (line 256) has no `id`
  and is preceded by the full command rail (30+ nav links) and topbar on every
  route. Keyboard users must Tab through the entire rail on each page.
- Fix: render `<SkipLink />` as the first element in App's product-shell return
  (before `<LaunchScreen />` / `<Rail />`, around line 249) and give main a
  focusable id:
  ```jsx
  import SkipLink from './components/SkipLink.jsx';
  // ...
  return (<div><SkipLink /> ...
    <main id="main-content" tabIndex={-1} className="rl-content">
  ```
  Add the same `<SkipLink />` + `id="main-content"` on the `MarketingShell`
  branch (lines 222-241).
- WCAG: 2.4.1 Bypass Blocks.

### P1.5 Clickable non-button elements are not keyboard operable
- Where:
  - `src/components/UI.jsx` `Stat` onClick `<div>` (line 58) and `StatCard`
    onClick `<div>` (line 269) - these are the dashboard KPI tiles that navigate
    to filtered views.
  - `src/components/CommandK.jsx` result rows (covered in P1.3).
- Impact: KPI tiles that act as links cannot be reached or activated by keyboard.
- Fix: when `onClick` is present, render an interactive element:
  ```jsx
  <div role="button" tabIndex={0} onClick={onClick}
       onKeyDown={(e)=>{ if(e.key==='Enter'||e.key===' '){e.preventDefault();onClick(e);} }}
       aria-label={typeof label==='string' ? label : undefined} ...>
  ```
  Long term, prefer routing these through a real `<Link>`/`<button>`.
- WCAG: 2.1.1 Keyboard, 4.1.2.

## Priority 2 - color contrast (measured against `--ink`/`--paper` tokens)

Ratios computed against the light theme surfaces `--paper #ffffff` /
`--page #f6f7f9`. AA requires 4.5:1 for normal text, 3:1 for >=18.66px bold or
>=24px, and 3:1 for UI component boundaries.

### P2.1 `--n-400` used as text fails AA (about 2.6:1 on white)
- Where: `src/index.css` line 180 `.input::placeholder { color: var(--n-400); }`
  (`#98a1b0`), plus `--n-400` used for the search-hint icon and chevrons in
  `src/components/CommandK.jsx` (lines 75, 94) and the topbar search icon.
- Fix: darken placeholder text to `--n-600` (`#5b6474`, about 5.5:1) or introduce
  a `--placeholder: #6b7383` token. Decorative chevron/search glyphs can stay
  light if they are `aria-hidden` (they are, via `Icon`), but placeholder text
  is read as content and must pass.
- WCAG: 1.4.3 Contrast (Minimum).

### P2.2 `badge-warn` text on its own tint is borderline (about 3.9:1)
- Where: `src/index.css` line 153 `.badge-warn { color: var(--warn) #b3721a; background: var(--warn-bg) #fbefdb; }`. Badge text is `.78rem` (about 12.5px),
  not large, so it needs 4.5:1.
- Fix: darken `--warn` for text use to about `#8a5410`, or darken only inside the
  badge: `.badge-warn { color: #8a5410; }`. Re-check `badge-ok`/`badge-risk`
  (both pass) after any token change.
- WCAG: 1.4.3.

### P2.3 Nav section labels are dimmed below AA
- Where: `src/index.css` line 331 `.nav-sec` uses `--nav-muted #8b93a4` at
  `opacity: .7` and `.66rem` uppercase on `--nav #12151c`. The opacity drops the
  effective ratio under 4.5:1 for that small text.
- Fix: remove `opacity: .7` and use a slightly brighter muted (`#9aa2b3`), or
  accept it only if these are treated as decorative dividers (they are not - they
  label groups).
- WCAG: 1.4.3.

Note: `--n-600` body-muted text (`.muted`, `.stat-label`, table `th`) passes at
about 5.5:1, and `--nav-muted` nav links pass at about 5.7:1. Keep those.

## Priority 3 - forms, labels, and announcements

### P3.1 Field labels are not programmatically associated
- Where: `src/components/UI.jsx` `Field` (lines 109-117) renders `<label>` as a
  sibling of `children`, with no `htmlFor`, and `Input/Select/Textarea` (lines
  118-120) have no `id`. Same pattern in `src/gate/ComingSoon.jsx` (labels lines
  104-125 are not tied to their inputs).
- Impact: clicking a label does not focus its input; screen readers may not
  announce the field name.
- Fix: generate an id in `Field` and pass it down via context or a render prop:
  ```jsx
  export function Field({ label, hint, id: fixedId, children }) {
    const id = fixedId || useId();
    const child = React.isValidElement(children)
      ? React.cloneElement(children, { id, 'aria-describedby': hint ? id+'-hint' : undefined })
      : children;
    return (<div className="field">
      {label && <label htmlFor={id}>{label}</label>}
      {child}
      {hint && <span id={id+'-hint'} className="t-xs muted">{hint}</span>}
    </div>);
  }
  ```
  For ComingSoon, add `htmlFor`/`id` pairs (e.g. `id="cs-name"`).
- WCAG: 1.3.1 Info and Relationships, 3.3.2 Labels or Instructions, 4.1.2.

### P3.2 Toasts are invisible to screen readers
- Where: `src/components/UI.jsx` `useToast` (lines 289-298) appends a bare
  `<div>` to `body` with no `aria-live`/`role`. Confirmation and error messages
  are never announced.
- Fix: route toast text through the announcer in `src/lib/a11y.js`:
  ```jsx
  import { announce } from '../lib/a11y';
  // inside show(): announce(msg, { assertive: tone === 'risk' });
  ```
  The visual toast can stay; `announce` adds the aria-live channel.
- WCAG: 4.1.3 Status Messages.

### P3.3 Route changes are not announced
- Where: `src/App.jsx` route effect (line 214) scrolls to top but never tells
  assistive tech the page changed (SPA nav does not move focus or re-announce a
  title by default).
- Fix: use `useAnnouncer` on pathname change and move focus to `#main-content`:
  ```jsx
  import { useAnnouncer } from './lib/a11y';
  const say = useAnnouncer();
  useEffect(() => {
    document.getElementById('main-content')?.focus();
    say(document.title || 'Page loaded');
  }, [loc.pathname]);
  ```
- WCAG: 4.1.3, 2.4.3.

## Priority 4 - landmarks, motion, and polish

### P4.1 Unlabeled landmark regions
- Where: `src/App.jsx` `<aside className="rl-rail">` (line 115) and its inner
  `<nav>` (line 130) have no `aria-label`. With marketing nav elsewhere, multiple
  unlabeled navs are ambiguous to screen-reader landmark navigation.
- Fix: `<aside aria-label="Primary">` and `<nav aria-label="Main">`. Add
  `aria-label="Primary"` to the marketing shell nav too.
- WCAG: 1.3.1, 2.4.1.

### P4.2 JS-driven animation ignores reduced-motion
- Where: `src/components/UI.jsx` `useCountUp` (lines 8-20) and `AnimatedNumber`
  (lines 24-38) run rAF tweens unconditionally; `Sparkline`/`Ring`/`ProgressBar`
  use CSS transitions the `@media (prefers-reduced-motion)` block (index.css
  lines 278-282) does not cover. The CSS block handles ambient/page/stat
  animations well, but not these JS counters.
- Fix: gate the tween with `usePrefersReducedMotion` from `src/lib/a11y.js`:
  ```jsx
  import { usePrefersReducedMotion } from '../lib/a11y';
  const reduce = usePrefersReducedMotion();
  useEffect(() => { if (reduce) { setN(value); return; } ...rAF... }, [value, reduce]);
  ```
- WCAG: 2.3.3 Animation from Interactions (AAA, but commonly audited), 2.2.2.

### P4.3 Data visualizations have no text alternative
- Where: recharts charts in `src/pages/Reports.jsx` (lines 8-11 import),
  `Dashboards.jsx`, `Forecasting.jsx`, `Invoices.jsx`, plus decorative
  `Sparkline`/`MiniBars`/`Ring` in `UI.jsx` (lines 194-220, 178-191).
- Fix: give each meaningful chart a `role="img"` + `aria-label` summarizing the
  trend (for example, "Pipeline by stage, Discovery highest at 1.2M"), and mark
  purely decorative sparklines `aria-hidden="true"`. Pair every chart with the
  data table Rally already renders (Reports does) so the numbers are readable.
- WCAG: 1.1.1 Non-text Content.

### P4.4 Icon-only controls - already correct, keep it that way
- `src/App.jsx` topbar buttons (lines 175, 185-193) and Rail close (line 126) all
  carry `aria-label`, and `Icon` sets `aria-hidden="true"` (icons.jsx line 75).
  This is the standard to hold new icon buttons to. The Modal close uses
  `aria-label="Close"` (UI.jsx line 138) - good.

## Suggested verification pass

1. Automated: run axe-core (or Lighthouse a11y) on `/app`, `/deals`,
   `/reports`, and the marketing `/`. Expect P1.1, P1.5, P2.1 to clear the most
   findings.
2. Keyboard only: Tab from page load - the skip link should appear first, then
   focus should reach main content; open Cmd+K and a modal and confirm focus is
   trapped and returns on Escape.
3. Screen reader (NVDA/VoiceOver): confirm route changes and toasts are
   announced, form fields read their labels, and charts expose a summary.
